- [x] **BUG5 — HIGH — correctness/security (scope escalation)** — FIXED 2026-06-13 (option 1: `owner` on B2B/Customer resources, `payment:write` dropped from web clients + vite) — `Config.cs:117-118` + `Concertable.Payment.Web/Program.cs:90` + `RoutingExtensions.cs:11-13`
  Adding `payment:write` to the **interactive web clients'** `AllowedScopes` lets any logged-in user mint an access token carrying `scope: payment:write` — which is the *exact and only* claim Payment's `ServiceToken` policy checks, and that policy is what guards the service-to-service gRPC surface (`EscrowGrpcService`, `ManagerPaymentGrpcService`, `CustomerPaymentGrpcService`), whose requests take payer/payee ids as plain parameters. A customer or venue manager can therefore call escrow capture/deposit/release or `ManagerPayment.Pay` directly with arbitrary owner ids, using a public PKCE client (the attacker doesn't even need the SPA to request the scope — crafting the authorize URL with `scope=payment:write` suffices, since `AllowedScopes` is the only gate). The motivation is clear — `owner` is a `UserClaims` of the `concertable.payment.api` ApiResource, and `payment:write` is that resource's only scope, so requesting it is currently the only way to get the `owner` claim into user tokens for `StripeAccountController`. Fix without the escalation: emit `owner` from the b2b/customer ApiResources (Payment already accepts those audiences — `Program.cs:84`), or add a separate user-grade ApiScope (e.g. `payment:user`) to the payment ApiResource and key the user endpoints' authorization on that, keeping `payment:write` client-credentials-only. Note also the mobile clients were *not* given the scope, so mobile tokens carry no `owner` claim and every `ICurrentPayoutOwner` endpoint now throws `UnauthorizedException` for them (pre-branch these keyed off `ICurrentUser.GetId()`, which any token satisfied) — worth deciding whether mobile is in scope before shipping.
