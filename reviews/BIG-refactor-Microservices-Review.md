## Stage 1 — Shared foundation — reviewed 2026-06-03

### Correctness (high → low)

- [x] **BUG1 — HIGH — correctness** — `api/Concertable.Messaging/Concertable.Messaging.AzureServiceBus/AzureServiceBusTransport.cs:36`
  `PublishAsync` wraps `SendMessageAsync` in `catch (ServiceBusException) when (ex.Reason == MessagingEntityNotFound) { }` — an **empty** catch (no log, no rethrow). The outbox dispatcher (`OutboxDispatcher.DrainOnceAsync`) treats the returned task as success and calls `row.MarkDispatched(...)`, so a missing/mis-prefixed topic causes the integration event to be **permanently lost with zero observability**; Customer/Search never receive it. `SendAsync` (commands) has no equivalent swallow — the asymmetry shows this isn't a deliberate global policy. Fix: at minimum log + surface; ideally let it fail so the row stays Pending and retries.

- [x] **BUG2 — HIGH — correctness** — `api/Concertable.Messaging/Concertable.Messaging.Infrastructure/Outbox/OutboxReader.cs:18` (with `OutboxDispatcher.cs:56`)
  `GetPendingAsync` selects `Status == Pending` rows and `Take(batchSize)` with **no claim/lock** — no `Status -> Dispatching` transition, no `UPDLOCK`/`READPAST`. Every `OutboxDispatcher` instance polling the same DB reads the same batch, publishes it, and only saves `Dispatched` at the end of the batch. With B2B Web **and** Workers both registering the dispatcher (default `runDispatcher: true`) against `B2BDb` — or any scaled-out replica set — **every event is published 2+ times** each cycle. Mitigated only insofar as every consumer has inbox dedup. Fix: atomic claim (e.g. `UPDATE … OUTPUT` / row-version transition) or run the dispatcher in exactly one host.

- [x] **BUG3 — MEDIUM — correctness** — `api/Concertable.Messaging/Concertable.Messaging.AzureServiceBus/AzureServiceBusReceiver.cs:43,54`
  Processors are created via `client.CreateProcessor(topic, serviceName)` / `CreateProcessor(queue)` with **no `ServiceBusProcessorOptions`**, so `AutoCompleteMessages` defaults to `true` while the callback also settles manually (`CompleteMessageAsync` / `AbandonMessageAsync`). Any handler path that ever returns without settling (and without throwing) would be silently auto-completed = unprocessed message acked. Fix: pass `new ServiceBusProcessorOptions { AutoCompleteMessages = false }` since settlement is manual.

- [x] **BUG4 — MEDIUM — correctness** — `api/Concertable.Messaging/Concertable.Messaging.AzureServiceBus/AzureServiceBusReceiver.cs:110,155`
  On handler failure the catch calls `AbandonMessageAsync` (immediate re-queue, no delay). For a transient fault (DB outage) the same message is redelivered in a tight loop with `MaxConcurrentCalls` until `MaxDeliveryCount` is exhausted, then the broker dead-letters a message that would have succeeded once the dependency recovered. No transient-vs-permanent distinction / backoff.

- [x] **BUG5 — MEDIUM — correctness** — `api/Concertable.Messaging/Concertable.Messaging.Infrastructure/Outbox/LocalDispatchingBus.cs:27`
  In-process dispatch builds the envelope with `MessageType = typeof(TEvent).FullName!`, whereas the outbox dispatcher (`OutboxDispatcher.cs:69`, from `row.MessageType`) and the ASB receiver (`Receiver.cs:84`, from the `MessageType` app-property) use the canonical `[MessageType("…")]` short name. The **same event presents a different `envelope.MessageType` depending on whether a handler ran in-process or off the bus** — any handler that logs, branches on, or persists `MessageType` (e.g. into an inbox row) is inconsistent. Envelope `CorrelationId` is also hard-coded `null` here.

- [x] **BUG6 — MEDIUM — correctness (latent)** — `api/Shared/Concertable.Kernel/Auth/ClientCredentialsTokenService.cs:11,22`
  The token cache is a single `_cachedToken` keyed only by expiry, but `GetTokenAsync(string scope)` takes a scope. The fast-path (`line 22`) and double-checked path return the cached token **regardless of the requested scope**, so once a token for scope A is cached, a call for scope B returns A's token until expiry → downstream `403/insufficient_scope`. Latent today (callers use one scope) but the multi-scope public signature invites it. Fix: key the cache by scope.

### Stage 1 — Resolution (2026-06-03, all 10 fixed; build green, messaging unit tests pass bar the pre-existing BUG21 set)
- **BUG1** — dropped the empty `MessagingEntityNotFound` swallow in `AzureServiceBusTransport.PublishAsync`; failures now propagate so the outbox records the failure, retries, and logs.
- **BUG2** — atomic claim implemented. New `OutboxStatus.Dispatching` (int enum, no schema change); `OutboxReader.GetPendingAsync` claims via `WITH (READPAST, UPDLOCK, ROWLOCK)` + `UPDATE…OUTPUT` on SQL Server (LINQ fallback retained for the InMemory unit tests, which can't model row-locking). - **BUG3** — both `CreateProcessor` calls now pass `ServiceBusProcessorOptions { AutoCompleteMessages = false }`; all handler paths settle manually.
- **BUG4** — `AbandonWithBackoffAsync` replaces the immediate abandon: exponential delay (`2^(DeliveryCount-1)`, capped 30s, cancel-aware) before abandon; relies on the processor's default auto-lock-renewal.
- **BUG5** — `LocalDispatchingBus` now stamps the canonical `[MessageType]` name via `MessageTypeAttribute.Resolve`, matching the outbox dispatcher / ASB receiver.
- **BUG6 + CV1** — `ClientCredentialsTokenService` cache is now a `ConcurrentDictionary` keyed by scope (no cross-scope leak); underscore fields renamed.
- **BUG7** — outbox drain order now `.ThenBy(m => m.Id)` for determinism.
- **BUG8** — `Guid.TryParse` on `MessageId` hoisted out of the deserialize try; invalid id dead-letters as `InvalidMessageId` (new `Log.DeadLetteringInvalidMessageId`) instead of mislabelled `DeserializationFailed`.
- **MB1** — dropped the unused `Concertable.Seed.Shared` ProjectReference from `Concertable.DataAccess.Infrastructure.csproj` (verified no transitive consumer breaks — full solution build green).

### Correctness (high → low)

- [x] **BUG10 — HIGH — correctness (atomicity)** — `api/Concertable.Customer/Modules/Ticket/Concertable.Customer.Ticket.Infrastructure/Services/TicketService.cs:105-108` (`CompleteAsync`)
  The transaction (`unitOfWork.BeginTransactionAsync()`) is opened on **`TicketDbContext`**, but `concert.DecrementAvailability(quantity)` + `concertRepository.SaveChangesAsync()` (line 106) commit a **separate `ConcertDbContext`** (different context/connection, not enrolled in the ticket tx) *before* `unitOfWork.SaveChangesAsync()` / `tx.CommitAsync()`. ### Correctness (high → low)

- [ ] **BUG13 — HIGH — correctness** — `api/Concertable.Payment/Concertable.Payment.Infrastructure/Services/Webhook/WebhookProcessor.cs:94-97` (with `WebhookController.cs:22-37`, `WebhookQueue.cs:19-27`)
  `ProcessAsync` wraps its **entire** body — the `payment_intent.succeeded` → `PublishAsync(PaymentSucceededEvent)` and the closing `context.SaveChangesAsync` — in `catch (Exception ex) { logger.StripeWebhookProcessingError(...); }` (log-and-return, no rethrow). It runs **fire-and-forget** on the in-memory `IBackgroundTaskQueue` (`WebhookQueue.EnqueueAsync`), and `WebhookController.HandleWebhook` has **already returned `200 OK`** to Stripe before the work runs. So if `PublishAsync` or `SaveChangesAsync` throws on a transient DB/ASB fault, the dedup `StripeEventEntity` and the outbox row are never persisted, the exception is swallowed, and **Stripe never retries** (it got a 200). A genuinely-succeeded payment's `PaymentSucceededEvent` is permanently lost → no ticket / no settlement recorded — on an adapter whose whole job is "only emit for live webhooks." (Same swallow-the-event failure mode as Stage 1's BUG1, different ingestion path.) Fix: don't swallow (let the queue surface/retry it), and persist before acking Stripe — or move ingestion onto a durable retrying queue.

- [ ] **BUG14 — HIGH — correctness / security (regression)** — `api/Concertable.Auth/Services/AuthService.cs:87-101` (`VerifyEmailAsync`) and `:118-132` (`ResetPasswordAsync`)
  Both validate the token solely via `tokenEntity.IsActive`, which (`PasswordResetTokenEntity.cs:14` / `EmailVerificationTokenEntity.cs:14`) is **only** an expiry check (`DateTime.UtcNow < Expires`). On success neither method deletes nor marks the token, and the entities have **no consumed/used field** — so a password-reset token stays valid for its full 1-hour window and an email-verification token for 24 hours, and is **replayable** (an attacker who later obtains a spent reset link can reset the password again). - [ ] **BUG15 — MEDIUM — correctness** — `api/Concertable.Payment/Concertable.Payment.Infrastructure/EscrowService.cs:95-110` (`CaptureAsync`)
  `EscrowEntity.Create(...)` + `AddAsync` + `SaveChangesAsync` (lines 95-97) run **before** `paymentManager.CaptureAsync` (99). - [ ] **BUG16 — MEDIUM — correctness (robustness)** — `api/Concertable.Search/Concertable.Search.Infrastructure/Data/Configurations/ConcertReadModelConfiguration.cs:17-25` (with `ConcertProjectionHandler.cs:37-54`)
  The Search `Concerts` read-model declares **enforced FKs** `Concerts → Artists (ArtistId)` and `Concerts → Venues (VenueId)` (realized as real DB constraints in the `InitialCreate` migration), but the three read-models are populated by **independent** projection handlers driven by **separate, independently-ordered ASB subscriptions** (`ArtistChangedEvent`/`VenueChangedEvent`/`ConcertChangedEvent`). `ConcertProjectionHandler` inserts a `ConcertReadModel` with `ArtistId`/`VenueId` (line 41-43) with no guard that the parent rows exist. - [ ] **BUG17 — MEDIUM — correctness / security (regression)** — `api/Concertable.Auth/Services/AuthService.cs:79` (email verification) and `:108` (password reset)
